# H5 埋点监控与数据看板 — 开发计划

> 创建时间：2026-04-22
> 状态：待开发

---

## 一、需求背景

为 H5 端新增埋点监控功能，实现以下目标：

1. **用户行为监控** — 收集各页面访问量、用户点击次数
2. **转化率追踪** — 追踪淘口令获取转化率
3. **使用频次分析** — 分析各页面的访问频次和使用情况
4. **数据可视化** — 在管理端 Dashboard 页面呈现以上所有数据

**核心约束**：
- 不能影响 H5 页面的使用和访问体验
- 不能做侵入式代码调整（不改动现有业务逻辑）
- 埋点上报必须是非阻塞的，不影响用户操作

## 三、埋点事件设计

### 3.1 事件清单

| 事件 ID | 触发时机 | 触发位置 | 携带数据 |
|---|---|---|---|
| `page_view` | 路由变化（自动） | `App.tsx` | `page`, `referrer`, `duration` |
| `product_click` | 点击商品卡片 | `ProductsPage.tsx` | `product_id`, `product_name`, `position` |
| `tkl_generate` | 点击"获取淘口令" | `ProductDetail.tsx` | `product_id`, `platform_id`, `success` |
| `tkl_copy` | 淘口令复制成功 | `ProductDetail.tsx` | `product_id`, `platform_id` |
| `want_guzi_submit` | 求谷表单提交成功 | `WantGuzi.tsx` | `ip_name`, `has_category`, `has_remark` |
| `search` | 搜索框触发 | `ProductsPage.tsx` | `keyword`, `results_count` |
| `filter_apply` | IP标签/分类筛选 | `ProductsPage.tsx` | `filter_type`, `filter_value` |
| `favorite_toggle` | 收藏/取消收藏 | `ProductDetail.tsx` | `product_id`, `action`（add/remove）|
| `share_link` | 点击分享 | `ProductDetail.tsx` | `product_id` |

### 3.2 事件通用字段

每个事件统一携带以下字段：

```typescript
interface AnalyticsEvent {
  event_id: string;       // 客户端生成 UUID
  event_type: string;     // 事件 ID
  page: string;           // 当前页面路径
  timestamp: string;      // ISO 时间戳
  session_id: string;     // 会话 ID（页面加载时生成，存在 sessionStorage）
  user_agent: string;     // UA 字符串
  data: Record<string, any>; // 事件特有数据
}
```

## 四、数据看板指标体系

### 4.1 核心指标（KPI 卡片）

| 指标 | 计算方式 | 说明 |
|---|---|---|
| 今日 UV | 去重 session_id 数量 | 独立访客数 |
| 今日 PV | 所有 page_view 事件数 | 页面访问量 |
| 淘口令转化率 | 淘口令获取次数 / 商品详情页 PV | 核心业务转化 |
| 今日求谷提交 | want_guzi_submit 事件数 | 用户需求信号 |
| 今日商品点击 | product_click 事件数 | 用户兴趣指标 |
| 淘口令获取次数 | tkl_copy 事件数（成功复制） | 直接转化行为 |

### 4.2 图表模块

| 模块 | 图表类型 | 数据维度 |
|---|---|---|
| 访问趋势 | 折线图 | 按天，PV/UV 双线 |
| 淘口令趋势 | 折线图 | 按天，生成次数/复制次数 |
| 求谷趋势 | 折线图 | 按天，提交数 |
| 页面访问占比 | 饼图 | 各页面 PV 占比 |
| 页面访问频次 | 柱状图 | 各页面 PV 降序排行 |
| 热门商品点击 | 横向柱状图 | product_id + 次数 |
| 热门淘口令商品 | 横向柱状图 | product_id + 获取次数 |
| 热门搜索词 | 词云 / 排行榜 | keyword + 次数 |
| 转化漏斗 | 漏斗图 | 访问→点击→复制→求谷 |

### 4.3 时间范围

- 默认展示近 **7 天**数据
- 支持切换为 **30 天**
- Dashboard 页面数据 **手动刷新**，不做自动轮询

## 五、技术方案

### 5.1 H5 端实现

**新建文件：**
- `h5/src/utils/analytics.ts` — 埋点 SDK 核心模块

**修改文件（最小改动）：**

| 文件 | 改动说明 |
|---|---|
| `h5/src/App.tsx` | 在 `AppContent` 中加 `useLocation` effect 触发 `page_view`；离开时计算停留时长 |
| `h5/src/pages/products/ProductsPage.tsx` | 在 `renderProductCard` 的 `onClick` 中加 `track('product_click', ...)`；搜索和筛选处加 `track()` |
| `h5/src/pages/products/ProductDetail.tsx` | 在 `handleGetTkl`、`handleCopyTkl`、`handleToggleFavorite`、`handleShare` 中各加一行 `track()` |
| `h5/src/pages/products/WantGuzi.tsx` | 在 `submitWantGuzi` 成功后加 `track()` |
| `h5/src/api/config.ts` | 确认 API base URL 配置正确 |

**上报机制：**
- 优先使用 `navigator.sendBeacon`（非阻塞）
- 备选 `navigator.clipboard.writeText` 无法使用时用 `fetch` async 兜底
- 本地缓冲：上报失败时存入 `localStorage`，下次成功时重发

### 5.2 Server 端实现

**新建文件：**
- `server/app/database/analytics_dao.py` — MongoDB 增删改查
- `server/app/models/analytics.py` — Pydantic 模型
- `server/app/api/analytics.py` — API 路由

**新建集合：**
- `analytics_events` — 原始事件（按时间分表或设置 TTL 索引，自动过期旧数据，建议保留 90 天）
- `analytics_daily` — 每日聚合（由定时任务从原始事件聚合写入）

**API 接口：**

```
POST /api/v1/analytics/events
  Body: { events: AnalyticsEvent[] }   // 支持批量上报
  Response: { success: true, count: number }

GET /api/v1/analytics/stats
  Query: start_date, end_date, granularity(day)
  Response: { uv, pv, tkl_count, tkl_copy_count, want_guzi_count, ... }

GET /api/v1/analytics/trend
  Query: start_date, end_date, metrics[]
  Response: { dates: string[], series: { name, data: number[] }[] }

GET /api/v1/analytics/page-stats
  Query: start_date, end_date
  Response: { pages: { page, pv, uv, avg_duration }[] }

GET /api/v1/analytics/product-stats
  Query: start_date, end_date, limit
  Response: { clicks: [], tkl_copies: [] }

GET /api/v1/analytics/keyword-stats
  Query: start_date, end_date, limit
  Response: { keywords: { keyword, count }[] }
```

**定时任务：**
- 每小时运行一次聚合脚本，将 `analytics_events` 聚合写入 `analytics_daily`
- Dashboard 查询优先读 `analytics_daily`，保证查询性能

## 七、注意事项

1. **隐私合规**：埋点数据仅包含脱敏的行为数据（不含用户账号、位置等敏感信息）
2. **性能影响**：上报逻辑必须在主线程外执行，`sendBeacon` 即用即发，不等待响应
3. **数据量预估**：H5 端日均 PV 预估 5000~20000，事件数据存储 90 天约 100~500 万条文档，合理设置 MongoDB 索引
4. **Dashboard 性能**：Dashboard 查询必须走 `analytics_daily` 聚合表，禁止直接扫描 `analytics_events` 集合
5. **降级策略**：埋点 SDK 上报失败时静默吞掉异常，不向用户暴露任何错误提示

## 八、验收标准

- [ ] H5 页面加载时间无明显增加（增量 < 5ms）
- [ ] 用户操作无感知，埋点调用不阻塞任何交互
- [ ] Dashboard 各图表数据与实际用户行为一致
- [ ] 淘口令转化率 = `tkl_copy` 次数 / `page_view(product_detail)` 次数，数值合理
- [ ] 历史数据可查询（默认展示近 7 天）
- [ ] MongoDB TTL 自动清理 90 天前的事件数据
